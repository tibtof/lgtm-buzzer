# LGTM-Buzzer ‚Äî Project Constitution

## What this project is

LGTM-Buzzer is a browser extension that intercepts the "Approve" button on
GitHub and Azure DevOps pull requests and gates it behind a short quiz about
the actual diff. The quiz is generated by a local LLM CLI (Claude Code, Codex,
or `gh copilot`) reached through a native messaging host. If you can answer
the quiz, the approval goes through. If you can't, you didn't read the PR.

Target: software engineers who want to keep their teammates (and themselves)
honest about reading the diff before approving.

License: MIT

## Locked decisions (do not relitigate)

| Decision | Choice | Reason |
|---|---|---|
| Language | TypeScript (strict) | Shared types across host + extension; ecosystem fit for MV3 |
| Module system | ESM only | MV3 service workers are ESM; no CJS interop |
| Workspaces | npm workspaces | Stdlib of the ecosystem, zero extra tooling |
| Architecture | Hexagonal (ports & adapters) | Clean boundaries between domain, I/O, and UI |
| Domain style | Functional, pure-where-possible | Discriminated unions, immutable values, Result<T, E> |
| Error model | `Result<T, E>` over throwing | Throws reserved for programmer errors, never expected failures |
| Core deps | Zero runtime deps; zero Node, zero DOM | Core must be runnable in any JS env, embeddable in tests |
| Initial browser | Chrome (MV3) | Largest user base, drives the constraints |
| Later browser | Safari | Wraps the MV3 extension via the Xcode converter |
| LLM integration | Local CLI via native messaging | No remote calls from the extension; user keeps their LLM choice |
| LLM providers (v1) | Claude Code, Codex, `gh copilot` | All three are CLI-first and locally invoked |
| Quiz input | The actual diff, never the PR description/title | Prevents "LGTM-by-marketing" ‚Äî see Key differentiator |
| Distribution | Browser stores + the native host as a separate installable | Stores can't ship native binaries |

## Architecture principles

- **Hexagonal architecture, enforced by package boundaries.** Unlike a single
  flat package, here the boundary is the workspace import graph. Lints/tests
  must fail if `core` reaches into `adapters`, `host`, or `extension`.
- **Core is pure.** No `node:*`, no `fetch`, no `chrome.*`, no `globalThis`
  side effects. Only depends on `protocol`. Must be runnable in a plain JS
  sandbox.
- **Ports live with the domain, adapters live outside.** Every external
  capability the core needs (LLM, VCS, storage, clock, randomness) is a
  port defined in `core`. Adapters in `adapters/` implement those ports.
- **SOLID, functional bias.** Prefer pure functions and small types over
  classes. Use classes only when identity or lifecycle genuinely matters.
- **No global state.** Everything passes through constructor injection or
  explicit context arguments. No module-level singletons.
- **Discriminated unions over enums + booleans.** Make illegal states
  unrepresentable.
- **Result<T, E>, not throw.** Reserve `throw` for programmer errors
  (invariant violations, null where impossible). All expected failures
  return a typed `Err`.

### Package layout

```
packages/
  protocol/    Shared types for the stdio JSON protocol + domain DTOs.
               Zero deps. Imported by everyone.
  core/        Pure domain: QuizSession, ReviewGate, ports
               (LLMProvider, VCSProvider, QuizPolicy). Zero I/O,
               zero Node/DOM deps. May import from `protocol` only.
  adapters/    Concrete implementations of core ports:
               claude-cli, codex-cli, copilot-cli, github, ado.
               One subfolder per adapter. May import from
               `protocol` and `core` only.
  host/        Native messaging host. Thin wiring of adapters into
               core. Node-only. May import from anywhere except
               `extension`.
  extension/   MV3 extension: service worker, content scripts, UI.
               Browser-only. May import from `protocol` and `core`.
               MUST NOT import from `host` or `adapters` directly.
```

### Dependency-direction rule (hard)

```
protocol  ‚Üê core  ‚Üê adapters  ‚Üê host
protocol  ‚Üê core  ‚Üê extension
```

Arrows point in the allowed import direction. Anything else fails review.

For the current file inventory, run `ls packages/*/src` ‚Äî this document
describes principles, not file census.

## Dependency rules

- **`protocol`**: zero runtime deps. Types only. No `zod`/`io-ts` unless an
  ADR justifies it; prefer hand-written discriminated unions.
- **`core`**: zero runtime deps. No Node, no DOM. If you reach for a
  dependency in `core`, you are almost certainly building an adapter in
  disguise ‚Äî stop and write the port instead.
- **`adapters`**: may depend on Node stdlib and on adapter-specific libs
  (e.g., `@octokit/*` for the GitHub adapter). One dep set per adapter; do
  not pollute siblings.
- **`host`**: minimal Node deps for stdio framing.
- **`extension`**: browser/MV3 only; UI deps live here. No Node-only
  packages.
- **Approved licenses**: MIT, Apache-2.0, BSD-2/3, ISC.
- **Rejected**: GPL, AGPL, LGPL, CC-BY-SA, proprietary.
- New runtime deps in `core` or `protocol` require an ADR.

## Code style

- TypeScript `strict: true`, plus `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`.
- No `any`. Use `unknown` at boundaries and narrow.
- Prefer `type` aliases and discriminated unions over `interface` and `enum`.
- `readonly` on fields and arrays by default; mutate only inside a clearly
  scoped builder.
- `const` over `let` where the binding doesn't change.
- Optional chaining `?.` and nullish coalescing `??` over nested `&&`/`||`.
- Public functions/types: TSDoc with one short paragraph + `@param`/`@returns`
  when not obvious from the signature.
- No default exports ‚Äî named exports only. Easier to grep, refactor, and
  re-export.
- ESM imports use explicit `.js` extensions where the TS config requires it.
- Errors carry a discriminated `kind` field and structured context, not
  string-only messages.
- No `console.log` in committed code outside the `host` entry and dev-only
  builds; use a small logger port instead.

### Result<T, E> convention

```ts
export type Result<T, E> =
  | { ok: true; value: T }
  | { ok: false; error: E };

export const ok  = <T>(value: T): Result<T, never>  => ({ ok: true,  value });
export const err = <E>(error: E): Result<never, E> => ({ ok: false, error });
```

Use a single `Result` type across the codebase, defined in `protocol`. Every
expected failure path returns `Result`. Async work returns `Promise<Result<‚Ä¶>>`,
not a rejected promise.

## Agent pipeline

The standard flow for any GitHub issue:

```
PM Agent ‚Üí Architect Agent ‚Üí Dev Agent ‚Üí Reviewer Agent ‚Üí (your PR review) ‚Üí repeat until merged
```

- Always sequential. Each stage depends on the previous stage's output.
- State is passed between stages via GitHub issue status comments
  (`Status: READY_FOR_X`), not via memory or local files.
- The architect writes a full ADR to `decisions.md`; the issue thread only
  gets a short pointer comment.

## Status-machine vocabulary

Issues move through these states via comments:

- `READY_FOR_ARCH` ‚Äî PM has finalized the spec.
- `NEEDS_CLARIFICATION` ‚Äî architect bounced it back; PM owns the next move.
- `READY_FOR_DEV` ‚Äî architect's ADR is in `decisions.md`.
- `READY_FOR_REVIEW` ‚Äî dev pushed a PR.
- `CHANGES_REQUESTED` ‚Äî reviewer found blockers; dev owns the next move.
- `APPROVED` ‚Äî ready for human review.

## Sub-agent routing rules

**Sequential dispatch** (this project always uses sequential):
- PM ‚Üí Architect ‚Üí Dev ‚Üí Reviewer pipeline is always sequential.
- Each stage depends on the output of the previous one.

**Do not parallelize** stages ‚Äî they share `decisions.md`, branch, and PR
state.

**Background dispatch** is fine for:
- Research tasks (looking up library docs, license checks, MV3 API behavior).
- Codebase exploration that doesn't modify files.

## Key differentiator

**The quiz is generated from the diff, never from the PR description, title,
or commit messages.** A teammate writing a great PR description must not be
able to short-circuit the gate. Treat any path that feeds non-diff text into
the LLM prompt with the same care as a security boundary ‚Äî it can be used to
launder rubber-stamping past the quiz.

A close second: **the LLM stays local.** No remote model calls from the
extension or host. The native host shells out to the user's chosen CLI, which
the user already trusts. We never ship our own keys, never proxy through a
hosted service, never add telemetry that includes diff content.
